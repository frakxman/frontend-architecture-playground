Angular 17+

Signals vs Observables

Interact with both APIs side by side and see how each handles reactivity, derived state, and lifecycle cleanup.

Both counters do the same thing. Notice how Signals use synchronous reads with signal(), while Observables require the async pipe.

⚡

Signal

{{ sigRenders() }} renders
{{ sigCount() }}
count = signal(0);

increment() {
  this.count.update(n => n + 1);
}

// Template:
{{ count() }}
  • Synchronous read — no pipe needed
  • Fine-grained change detection
  • No subscription management
🔄

Observable

{{ obsRenders }} renders
{{ obsCount$ | async }}
count$ = new BehaviorSubject(0);

increment() {
  this.count$.next(
    this.count$.value + 1
  );
}

// Template:
{{ count$ | async }}
  • Requires async pipe or manual subscribe
  • Powerful operators (debounce, switchMap…)
  • Must unsubscribe to avoid leaks
💡

Signals trigger targeted re-renders — only the components that read the signal update. Observables with the async pipe trigger Zone.js change detection on the whole subtree.

Derived state stays in sync automatically. computed() is lazy and memoized — it only recalculates when its dependencies change.

⚡

Signal + computed()

{{ sigCount() }}
double {{ sigDouble() }}
message {{ sigMessage() }} 
count  = signal(0);
name   = signal('Angular');

double  = computed(() =>
  this.count() * 2
);
message = computed(() =>
  `Hello, ${this.name()}!
   Count: ${this.count()}`
);
  • Lazy — computed only when read
  • Memoized — skips equal values
  • Tracks dependencies automatically
🔄

BehaviorSubject + pipe()

{{ obsCount$ | async }}
double {{ obsDouble$ | async }}
message {{ obsMessage$ | async }} 
count$ = new BehaviorSubject(0);
name$  = new BehaviorSubject('Angular');

double$  = this.count$.pipe(
  map(n => n * 2)
);
message$ = combineLatest([
  this.count$, this.name$
]).pipe(
  map(([c, n]) =>
    `Hello, ${n}! Count: ${c}`
  )
);
  • Requires combineLatest for multiple sources
  • Rich operator library (RxJS)
  • Each pipe creates a new subscription
💡

computed() automatically tracks which signals it reads. With Observables you have to explicitly combine sources using combineLatest, withLatestFrom, etc.

One of the biggest ergonomic wins with Signals is automatic cleanup. Effects tear themselves down when the component is destroyed.

⚡

effect() — auto cleanup

export class MyComponent {
  count = signal(0);

  constructor() {
    // Automatically destroyed
    // with the component.
    // No OnDestroy needed.
    effect(() => {
      console.log(
        'count changed:',
        this.count()
      );
    });
  }
}
  • No OnDestroy needed
  • No risk of memory leaks
  • Cleanup is implicit and guaranteed
✅ Zero boilerplate cleanup
🔄

subscribe() — manual cleanup

export class MyComponent
  implements OnDestroy {

  private sub = new Subscription();

  constructor(private svc: DataService) {
    this.sub.add(
      this.svc.data$.subscribe(val => {
        console.log('data:', val);
      })
    );
  }

  ngOnDestroy() {
    // Forget this → memory leak
    this.sub.unsubscribe();
  }
}
  • Requires explicit ngOnDestroy
  • Forgetting causes memory leaks
  • Alternatively use takeUntilDestroyed()
⚠️ Manual unsubscribe required
💡

Angular 16+ introduced takeUntilDestroyed() to simplify Observable cleanup, but Signals still have the edge — the cleanup contract is enforced by the framework, not by the developer.

Quick Reference

Feature ⚡ Signals 🔄 Observables
Read syntax count() count$ | async
Derived state computed() pipe(map(…))
Side effects effect() subscribe()
Cleanup Automatic Manual (unsubscribe)
Async streams Limited Native strength
Change detection Fine-grained Zone.js (default)
Learning curve Low Medium–High
