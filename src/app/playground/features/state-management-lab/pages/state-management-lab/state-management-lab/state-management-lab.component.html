State Management Patterns

Compare Services + RxJS vs modern Signal State. See how each manages data, derived state, and reactivity.

Both approaches manage a simple counter. RxJS requires the async pipe for template binding; Signals are read synchronously.

// service.ts
export class StateService {
  private counter = new BehaviorSubject(0);
  counter$ = this.counter.asObservable();

  increment() {
    this.counter.next(
      this.counter.value + 1
    );
  }
}

  • RxJS operators available

Signal State

{{ sigCounter() }}

synchronous read — no pipe

// component.ts
export class MyComponent {
  counter = signal(0);

  increment() {
    this.counter.update(n => n + 1);
  }
}

// template
{{ counter() }}
  • Direct component access
  • No service needed for simple state
  • Synchronous — easier to test
  • Fine-grained change detection
💡

Signals are colocated with the component, making simple state management straightforward. Services + RxJS are better for cross-component communication and when you need powerful stream operators.

Managing multiple related pieces of state. RxJS requires explicit combination; Signals use separate instances and auto-track dependencies.

profile$ = combineLatest([
  this.counter$,

  • Use combineLatest, withLatestFrom, merge
  • Explicit dependency management
  • Boilerplate for each combo

Separate Signals

{{ sigCounter() }} 
counter = signal(0);
name = signal('Angular');
email = signal('...');

profile = computed(() => {
  return {
    count: this.counter(),
    name: this.name(),
    email: this.email()
  };
});
  • Create signals as needed
  • Computed auto-tracks dependencies
  • No explicit combining required
  • Minimal boilerplate
💡

Signals don't require ceremony to combine multiple pieces. Each signal is independent, and computed() automatically knows which ones it reads. RxJS demands explicit channel merging via operators.

Deriving new state from sources. RxJS uses map/combineLatest; Signals use computed which is lazy, memoized, and auto-tracked.

  counter$, name$, email$

  map(([c, n, e]) =>

);
  • Each selector runs on every emission
  • Can add operators (debounce, throttle…)

  • Lazy — only runs when read
  • Memoized — skips equal values
  • Dependencies auto-tracked
  • No boilerplate operators
💡

computed() is lazy and memoized by default. It only recalculates when accessed and only if its dependencies changed. With RxJS, you must use shareReplay() for similar behavior.
